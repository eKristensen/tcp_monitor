# Code Review Findings

Generated by multi-angle automated review of the initial Rust rewrite.
All items are now resolved (✅).

---

## Critical / High

- ✅ **#1 Probe drain loop holds permit forever** (`server.rs`)
  No read timeout on the probe drain loop. A client that connects and never
  closes holds one of 128 semaphore slots indefinitely.

- ✅ **#2 Heartbeat and probe share one semaphore** (`server.rs`)
  128 concurrent probe connections exhaust the shared semaphore, refusing all
  heartbeat sessions. Ports now have independent limits.

- ✅ **#3 Unbounded metric label growth** (`metrics.rs`)
  Every unique peer name that connects permanently allocates label entries;
  `prometheus-client` has no removal API. Mitigated by tracking seen server
  peer names in `seen_server_peers: Mutex<HashSet<String>>` and capping at
  `MAX_UNIQUE_SERVER_PEERS = 512`. Connections from names beyond the cap are
  rejected with a warning before any labels are allocated.

- ✅ **#4 Handshake timeout promotes connection to full session** (`protocol.rs`)
  `recv_handshake` returned `Ok("unknown")` on timeout, admitting port
  scanners as full sessions. Now returns an error; caller closes the connection.

- ✅ **#5 Peer address change silently ignored on reload** (`client.rs`)
  `run_manager` checked peer name only. Changed host/port now triggers task
  restart.

---

## Medium

- ✅ **#6 Stale echo resets `consecutive_missed`** (`client.rs`)
  A late echo arriving after a timeout window was silently accepted, resetting
  the miss counter. `run_session` now saves `sent_seq` before incrementing and
  validates `echo.seq == sent_seq`; a mismatch is counted as a miss.

- ✅ **#7 Server parameters not hot-reloaded** (`main.rs`, `server.rs`)
  `recv_timeout` was fixed at startup. Server now receives config updates via
  a watch channel.
  ⏳ `bind`/`port` changes still require a restart (rebinding a live port is
  disruptive by design).

- ✅ **#8 No graceful shutdown** (`main.rs`)
  SIGTERM dropped all tasks abruptly, causing TCP RST instead of FIN on all
  peer connections. Shutdown now cancels tasks and awaits them.

- ✅ **#9 Zero config values not validated** (`config.rs`)
  `reconnect_delay = 0` caused a CPU-spinning reconnect loop;
  `heartbeat_interval = 0` made every heartbeat appear missed immediately.
  Minimum value of 1 is now enforced for all duration/count fields.

- ✅ **#10 `node.name` not updated on SIGHUP** (`main.rs`)
  Node name was captured once at startup. Client manager now reads it from
  the live config on each reload.

---

## Low / Maintenance

- ✅ **#11 `BrokenPipe` mapped to `"local_error"`** (`server.rs`)
  `EPIPE` during an echo write is a remote-initiated event; now maps to
  `"connection_reset"`.

- ✅ **#12 `next_send` advanced before echo wait** (`client.rs`)
  A timeout arms fires immediately at the next loop iteration because
  `next_send` was already advanced before waiting for the echo. After a timeout
  `next_send` is now reset to `Instant::now() + heartbeat_interval` so the
  next send is always spaced a full interval away.

- ✅ **#13 Echo re-serializes instead of sharing encoder** (`server.rs`)
  `drive_session` manually built the echo buffer. Extracted
  `protocol::encode_packet` shared by both sender and echo paths.

- ✅ **#14 `"unknown"` peer metric collision** (`server.rs`)
  Multiple handshake-failed connections shared `peer="unknown"`, corrupting
  gauges. Connections that fail or time out during handshake are now closed
  immediately without entering a session.

- ✅ **#15 `ClientConfig` changes not applied to running sessions** (`client.rs`)
  Already-running peer loops kept the old `heartbeat_interval`/`max_misses`
  until they next reconnected. `run_manager` now tracks `last_client_cfg` and
  cancels and respawns all peer tasks whenever `ClientConfig` changes on reload.
